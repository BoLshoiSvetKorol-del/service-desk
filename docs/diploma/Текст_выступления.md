# Текст выступления к презентации дипломного проекта
## «Корпоративная система управления заявками — Service Desk System»

> **Принцип:** текст НЕ повторяет слайд — он дополняет его контекстом, примерами и объяснением решений.

## Слайд 1 — Титульный

Добрый день, уважаемые члены комиссии!

Представляю дипломный проект — корпоративная система управления заявками технической поддержки, или «Service Desk».

Тема возникла из наблюдения за реальной ситуацией во многих организациях: когда у сотрудника ломается компьютер, пропадает доступ к программе или возникает любой другой технический вопрос — он не знает, куда обращаться. Кто-то пишет в мессенджер знакомому айтишнику, кто-то звонит, кто-то пишет на почту. В итоге заявки теряются, никто не знает кто за что отвечает, и проблема может висеть неделями. Именно эту проблему и решает данный проект.

---

## Слайд 2 — Цели и задачи

Давайте разберём, как этот процесс выглядит до автоматизации.

Сотрудник обнаружил проблему — он пишет письмо в IT-отдел. Письмо может прийти на общий ящик, где его увидят пять человек, и каждый подумает «другой разберётся». Или, наоборот, напишет конкретному специалисту, который в отпуске. Нет единого журнала обращений, нет приоритетов, нет сроков. Руководитель не может ответить на вопрос «сколько заявок сейчас в работе и когда они будут выполнены».

Из этой проблемы вытекает **цель** дипломного проекта — создать систему, которая наводит порядок в этом процессе.

Что касается **задач**, объясню каждую простым языком:

- *Проанализировать предметную область* — изучить, как устроена работа службы поддержки, какие роли существуют, какие данные нужно хранить.
- *Разработать REST API* — REST API это стандартный способ общения между браузером пользователя и сервером: браузер отправляет запрос «дай мне список заявок», сервер отвечает данными. Для реализации я использовал FastAPI — это современный Python-фреймворк, который автоматически создаёт документацию и работает очень быстро благодаря асинхронной обработке запросов.
- *Реализовать SLA-движок* — SLA расшифровывается как Service Level Agreement, соглашение об уровне сервиса. Это конкретные временны́е обязательства: критичная поломка должна быть устранена за 1 час, обычный запрос — за 8 часов. Система сама считает дедлайны и предупреждает, если время заканчивается.
- *Разработать интерфейс на React* — React это библиотека для создания интерактивных веб-страниц. Именно она отвечает за то, что страница обновляется без перезагрузки, таймер тикает в реальном времени, уведомления приходят мгновенно.
- *Реализовать уведомления через Celery* — Celery это система фоновых задач. Когда создаётся заявка, сервер не ждёт, пока письмо отправится, а сразу отвечает пользователю, а письмо уходит в фоне. Так система остаётся быстрой.

## Слайд 3 — Предметная область

На этом слайде показана суть проблемы и её решение в трёх блоках.

Хочу акцентировать внимание на среднем блоке — «Решение». Ключевое слово здесь — **единая точка входа**. Когда все заявки идут через одну систему, появляется прозрачность. Руководитель видит реальную картину: сколько заявок у каждого агента, где просрочки, какой отдел перегружен.

Ещё один важный момент — автоматическая маршрутизация. Когда пользователь создаёт заявку и выбирает тип «Проблема с принтером», система сама определяет, что это должен решать, например, отдел АХО, и направляет туда — без ручного вмешательства диспетчера.

---

## Слайд 4 — Целевая аудитория

На слайде три роли, хочу остановиться на ключевом принципе — **каждая роль видит только то, что ей нужно**.

Обычный сотрудник открывает систему и видит только свои заявки. Он не может случайно увидеть чужие проблемы или внутреннюю переписку службы поддержки.

Агент поддержки видит заявки своего отдела. Когда агент пишет внутренний комментарий — например, «нужно заказать запчасть, займёт 3 дня» — этот комментарий видят только коллеги-агенты, но не сотрудник, который подал заявку. Это позволяет вести рабочее обсуждение без лишней информации для клиента.

Администратор — это руководитель службы поддержки. Он видит полную картину и управляет настройками: добавляет новых сотрудников, создаёт типы заявок, смотрит отчёты по всей организации.

## Слайд 5 — Use Case диаграмма

Эта диаграмма — не просто картинка для презентации. Она была первым шагом проектирования.

Прежде чем писать код, нужно было ответить на вопросы: кто и что может делать в системе? Кто может менять статус заявки — только агент или и пользователь тоже? Кто может видеть отчёты? Может ли пользователь удалить свою заявку?

Use Case диаграмма формализует ответы на эти вопросы. Именно из неё потом вытекает ролевая защита в API: каждый эндпоинт знает, какая роль имеет к нему доступ, и при нарушении возвращает ошибку «Доступ запрещён».

---

## Слайд 6 — Sequence диаграмма

Эта диаграмма показывает важный архитектурный принцип — **не всё нужно делать сразу**.

Когда пользователь нажимает «Создать заявку», система должна ответить ему как можно быстрее. Но в этот момент нужно выполнить много операций: сохранить заявку в базу, рассчитать SLA-дедлайн, отправить два email (создателю и агенту), записать событие в историю.

Решение — разделить эти операции. Заявка сохраняется немедленно, пользователь сразу получает ответ. А отправка писем уходит в очередь задач Celery и выполняется в фоне, не заставляя пользователя ждать. Это сделало систему отзывчивой даже при большой нагрузке.

---

## Слайд 7 — Диаграмма состояний

Без строгого контроля статусов система превращается в хаос — заявку можно будет «выполнить», не взяв в работу, или снова открыть уже закрытую.

В системе реализована матрица допустимых переходов. Если кто-то попытается выполнить недопустимое действие через API — не через интерфейс, а напрямую — сервер откажет. Интерфейс вообще не показывает недопустимые кнопки.

Особо хочу отметить статус «Ожидает информации». Это единственный статус, при котором SLA-таймер ставится на паузу. Логика простая: если агент ждёт ответа от пользователя три дня — это не вина агента, и его нельзя наказывать нарушением SLA. Когда пользователь ответил и заявка переходит обратно в работу — таймер продолжается с того места, где остановился.

---

## Слайд 8 — Диаграмма деятельности

Здесь показано, что большая часть работы системы происходит **без участия человека**.

Пока агент работает с заявкой, фоновый процесс — Celery Beat — как будильник, каждые 10–15 минут просыпается и проверяет: есть ли заявки, у которых заканчивается время? Если у заявки осталось меньше часа до дедлайна — агент получает предупреждение прямо в интерфейс без перезагрузки страницы. Если дедлайн нарушен и прошло больше двух часов — уже руководитель отдела получает письмо об эскалации.

Это избавляет от необходимости вручную следить за сроками.

---

## Слайд 9 — Проектирование базы данных

База данных — это фундамент всей системы, и к её проектированию я подходил тщательно.

Хочу выделить несколько интересных решений. Во-первых, таблица **TicketHistory** — это неизменяемый журнал всех событий. Каждое действие с заявкой — смена статуса, назначение исполнителя, добавление комментария — записывается туда отдельной строкой и никогда не удаляется. Это даёт полную прозрачность: всегда можно восстановить, кто, когда и что сделал.

Во-вторых, гибкое хранение данных истории в формате **JSONB** — это специальный тип PostgreSQL, который позволяет хранить произвольную структуру данных прямо в ячейке таблицы. Например, для события «смена статуса» сохраняется `{"from": "new", "to": "in_progress"}`, а для события «изменение приоритета» — `{"from": "normal", "to": "high"}`. Не нужно создавать отдельную таблицу для каждого типа события.

Схема базы данных управляется через Alembic — систему миграций. Это как система контроля версий, только для структуры базы данных. Каждое изменение — отдельный файл миграции, который можно применить или откатить.

## Слайд 10 — Обоснование средств реализации

Хочу пояснить не сами технологии, а **почему именно они**, а не альтернативы.

Вместо FastAPI можно было взять Django — это тоже Python-фреймворк, более популярный и старый. Но Django создан для сайтов с шаблонами и страницами, а нам нужен только API. FastAPI изначально спроектирован для этого: он работает асинхронно, то есть один процесс может обслуживать тысячи запросов одновременно — это критично для SSE-уведомлений в реальном времени.

Вместо SSE можно было взять WebSocket — более мощный протокол двустороннего общения. Но в нашей задаче только сервер отправляет события клиенту, а не наоборот. SSE — это стандартный HTTP, он проще в реализации, автоматически переподключается при разрыве, и его не блокируют корпоративные фаерволы.

React вместо обычного JavaScript выбран, потому что система сложная — много форм, таблиц, фильтров, реальное время. React позволяет разбить интерфейс на независимые компоненты и переиспользовать их. Например, компонент «SLA-таймер» написан один раз и вставлен в пяти разных местах.

## Слайд 11 — Функционал системы

Здесь показаны реальные скриншоты работающей системы.

Обращу внимание на несколько деталей, которые важны с точки зрения пользовательского опыта.

На экране списка заявок — **цветные таймеры SLA**. Зелёный означает, что времени достаточно, жёлтый — меньше половины осталось, оранжевый — критически мало, красный — просрочено. Агент видит ситуацию одним взглядом, без необходимости считать что-либо.

На экране создания заявки — **интеллектуальное заполнение**: когда выбираешь тип заявки, система автоматически подставляет нужный отдел и показывает расчётный дедлайн — когда именно будет обработана заявка с учётом текущего времени и рабочих часов.

На экране отчётов — **экспорт в Excel**. Руководитель может выгрузить аналитику за любой период, отфильтровать по отделу или приоритету, и открыть в привычной Excel-таблице.

## Слайд 12 — Архитектура развёртывания

Система запускается в **Docker** — это технология контейнеризации. Представьте, что каждый компонент системы упакован в отдельную «коробку» со всем необходимым. Эти коробки запускаются вместе и общаются между собой по внутренней сети.

Такой подход решает классическую проблему «у меня работает, а у вас нет» — система будет вести себя одинаково на любом компьютере, где установлен Docker, будь то ноутбук разработчика или сервер в организации.

Nginx стоит перед всей системой как «охранник на входе»: запросы к `/api` он направляет на Python-бэкенд, запросы к обычным страницам — на React-приложение, файлы-вложения отдаёт напрямую, не нагружая Python.

## Слайд 13 — Тестирование

Тестирование — это не просто «проверить, что работает». Интереснее то, **что нашли в процессе**.

При тестировании SLA-движка обнаружился граничный случай: если заявка создаётся в пятницу в 17:45, а SLA — 4 рабочих часа, то дедлайн должен быть не в 21:45 пятницы, а в понедельник в 10:45. Без теста эту ошибку легко пропустить, и клиент бы получал некорректные дедлайны по всем заявкам, созданным в конце рабочей недели.

Ещё один важный аспект — тестирование ролевых ограничений. Каждый защищённый эндпоинт тестировался с тремя токенами: пользователя, агента и администратора. Это гарантирует, что никто не сможет получить чужие данные, даже если знает URL.

В итоге 47 автоматических тестов запускаются за несколько секунд и дают уверенность, что изменения в коде не сломали работающую логику.

## Возможные вопросы комиссии и ответы

**В: Почему выбрали FastAPI, а не Django?**
О: Django — это фреймворк «всё включено»: шаблоны страниц, встроенная админка, ORM. Для нашей задачи это лишнее, нам нужен только API. FastAPI легче, быстрее и изначально асинхронный — это важно для SSE-соединений, которых может быть одновременно много.

**В: Что такое Docker и зачем он нужен?**
О: Docker — технология контейнеризации. Каждый компонент системы упаковывается в изолированный контейнер со всеми зависимостями. Это решает проблему воспроизводимости: система будет работать одинаково на любой машине. Весь стек из шести компонентов поднимается одной командой `docker compose up`.

**В: Почему SSE, а не WebSocket?**
О: WebSocket — двусторонний канал, когда и сервер, и браузер могут отправлять данные. В нашем случае только сервер отправляет события — SSE для этого достаточно. Плюс SSE проще в реализации и работает через обычный HTTP, что важно в корпоративных сетях с фаерволами.

**В: Как обеспечивается безопасность?**
О: Несколько уровней. Во-первых, токены: access-токен живёт 15 минут — даже если перехватить, он быстро устаревает. При выходе refresh-токен блокируется. Во-вторых, ролевая защита: каждый запрос к API проверяет роль пользователя, нарушение — ошибка 403. В-третьих, файлы: при загрузке проверяется реальный тип файла, а не только расширение.

**В: Что такое SLA и как он рассчитывается?**
О: SLA — это временны́е обязательства. Например, критичная заявка должна быть решена за 1 рабочий час. «Рабочий» — это важно: если заявка создана в пятницу в 17:30, а SLA 4 часа — дедлайн будет не в 21:30, а в понедельник в 10:30. Система сама это считает. При статусе «Ожидает информации» таймер ставится на паузу — агент не наказывается за время ожидания ответа от пользователя.

**В: Есть ли мобильная версия?**
О: Отдельного мобильного приложения нет, но веб-интерфейс адаптивен — Ant Design автоматически подстраивает компоненты под ширину экрана. На телефоне система будет работать через браузер.

# Инструкция: запуск системы в вузе с флешки

## Что нужно на компьютере в вузе

**Обязательно:** установленный **Docker Desktop** (или Docker Engine на Linux).
Без Docker систему не запустить — нужно либо узнать заранее, есть ли он в аудитории, либо попросить установить.

Проверить можно командой:
```
docker --version
```
Если выдаёт версию — всё есть. Если «команда не найдена» — Docker не установлен.

---

## Вариант А — флешка с исходниками (если Docker есть и есть интернет)

Это самый простой вариант, но потребует скачивания образов при первом запуске (~1–2 ГБ).

**На своём компьютере:** скопировать на флешку папку `ProjectX` целиком.

**На компьютере в вузе:**
```bash
# 1. Перейти в папку проекта (путь к флешке — например E:\)
cd E:\ProjectX

# 2. Создать .env файл (скопировать из примера)
copy backend\.env.example backend\.env

# 3. Запустить все контейнеры
docker compose up --build -d

# 4. Дождаться запуска (30–60 секунд), затем открыть браузер:
#    http://localhost
```

Логин администратора: `admin` / пароль из `.env` (по умолчанию `admin123`).

---

## Вариант Б — флешка с готовыми образами (без интернета, рекомендуется)

Это надёжнее: образы уже собраны, интернет не нужен.

### Шаг 1 — на своём компьютере (один раз)

```bash
cd Z:\ProjectX

# Собрать все образы
docker compose build

# Сохранить образы в один файл (~800 МБ - 1.5 ГБ)
docker save projectx-backend projectx-frontend postgres:16 redis:7 nginx `
  | gzip > E:\ProjectX\docker_images.tar.gz
```

> Файл `docker_images.tar.gz` получится большим — убедитесь, что на флешке достаточно места (минимум 2 ГБ свободных).

### Шаг 2 — на компьютере в вузе

```bash
# 1. Загрузить образы из файла
docker load -i E:\ProjectX\docker_images.tar.gz

# 2. Перейти в папку проекта
cd E:\ProjectX

# 3. Создать .env
copy backend\.env.example backend\.env

# 4. Запустить (образы уже есть, сборка не нужна)
docker compose up -d

# 5. Открыть http://localhost
```

## Вариант В — без Docker (крайний случай)

Если Docker в вузе нет совсем — запустить можно только backend без контейнеров, но это сложнее и потребует установки Python, PostgreSQL и Redis вручную. Лучше договориться заранее о наличии Docker.

## Что делать если что-то пошло не так

```bash
# Посмотреть статус контейнеров
docker compose ps

# Посмотреть логи (если контейнер не запускается)
docker compose logs backend
docker compose logs postgres

# Перезапустить всё
docker compose restart

# Остановить и удалить (если нужно начать заново)
docker compose down
docker compose up -d
```
